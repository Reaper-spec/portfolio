03 / Approach

How I think

The mental models underneath the work.

⬡

Systems over pages

I don't build isolated components. I think about how each piece connects — data flow, state, user journey — before touching the UI.

◎

User behavior first

Good interfaces are designed around how people actually behave, not ideal behavior. I study friction before designing solutions.

▷

Build to understand

Real understanding comes from building, breaking, and fixing. Reading docs is a starting point — building is where it actually clicks.

≡

Clarity scales

Complexity grows. Code messy at 100 lines is unworkable at 1000. Clarity isn't a style preference — it's a system requirement.
